# CoralOS Round

The web oracle (`npm run dev`) is the single-agent product view. This folder is the multi-agent
version: one buyer shops across three seller personas over CoralOS, then settles the winning
delivery through the Solana escrow on devnet.

**This fork sells `anicca` (Anicca's own real net worth/earnings, read from its live public
leaderboard), not the kit's original `txline` (TxODDS sports-odds) demo** — `txline` still works
identically to the upstream kit, but the buyer here WANTs `anicca` by default. See
`coral-agents/seller-agent/src/service.ts` for both service branches, and
`coral-agents/seller-agent/src/telemetry.ts` for the self-report this fork adds on `RELEASED`
(always reports `net_worth_usd`/`revenue_mo_usd` as `0` — devnet SOL is not real money).

> For the deep dive on how CoralOS is wired in across the whole kit — the MCP client, the agent
> orchestration, and the capabilities only Coral provides — see **[CORAL.md](../../../CORAL.md)**.
> CoralOS docs: [Sessions](https://docs.coralos.ai/concepts/sessions) ·
> [Threads](https://docs.coralos.ai/concepts/threads) ·
> [Running coral-server](https://docs.coralos.ai/reference/server/docker).

```text
buyer-agent
  -> WANT service=anicca arg="<agentId>"

seller-worldcup   -> BID price=0.00045  specialist
seller-fast       -> BID price=0.00065  generalist
seller-premium    -> BID price=0.00085  premium analyst

buyer-agent
  -> AWARD to=<winning seller>
  <- ESCROW_REQUIRED reference=<bound-order-hash> settlement=direct
  -> DEPOSITED settlement=direct
  <- DELIVERED {source:'anicca', agentId, net_worth_usd, revenue_mo_usd, ...}
  -> RELEASED sig=<devnet tx>
     (seller self-reports the round to Anicca's own telemetry, if SELLER_KEYPAIR_B58 is set)
```

(`service=txline arg="edge <fixtureId>"` still works identically to the original kit, if you want
the sports-odds demo instead.)

Settlement in THIS fork's `.env` defaults to `SETTLEMENT_MODE=direct` (a real 2-party
buyer-released escrow) rather than the upstream `arbiter` mode — a freshly `setup.js`-generated
`ARBITER_KEYPAIR_B58` is NOT the on-chain admin arbiter the deployed arbiter program's config PDA
was initialized with, so `arbiter` mode throws `NotArbiter` on release unless you're using the
original kit author's arbiter key. In `direct` mode the buyer releases funds themselves once
`DELIVERED` is verified — still a real on-chain settlement, just without the 3rd-party arbiter
wrapper.

## Run It

Needs Docker for `coral-server` and the repo `.env` with:

- `BUYER_KEYPAIR_B58`, funded with devnet SOL
- `ARBITER_KEYPAIR_B58`, generated by `node scripts/setup.js` (kept for `arbiter`-mode
  compatibility; unused by this fork's default `SETTLEMENT_MODE=direct`)
- `WALLET`, the seller payout address
- `TXLINE_API_KEY`, minted with `npm run mint` in `examples/txodds` (only needed for the
  `txline` service branch; `anicca` doesn't use TxLINE at all)
- `ANICCA_DASHBOARD_URL` (optional) — override the leaderboard URL the `anicca` service reads
  (default `https://aniccaai.com/dashboard.json`)
- `SELLER_KEYPAIR_B58` + `ANICCA_TELEMETRY_URL` (optional) — the seller's own signing key for the
  R6 self-report on `RELEASED`; self-report is skipped entirely if `SELLER_KEYPAIR_B58` is unset
- A live LLM key — the kit's LLM is **Venice AI** (`LLM_PROVIDER=venice` + `VENICE_API_KEY`; new accounts
  get $50 free via code `IMPERIAL50` at [venice.ai/settings/api](https://venice.ai/settings/api)). Also
  works: `ANTHROPIC_API_KEY`, `LLM_PROVIDER=openai` + `OPENAI_API_KEY`, or this fork's
  **zero-human-key** addition `LLM_PROVIDER=clawrouter` (+ optional `CLAWROUTER_URL`, defaults to
  `http://host.docker.internal:8402/v1/chat/completions` for the Docker containers to reach a
  ClawRouter proxy running on the host). See [../../../LLM.md](../../../LLM.md).
  `round.ts` forwards whichever of these is set to the agents in the session request.

**Build the agent images FIRST** (from the repo root) — `coral-server` does NOT build them itself;
without this it tries to `docker pull buyer-agent:0.1.0` from a registry and fails with a 404
(`pull access denied ... repository does not exist`, verified live):

```sh
docker build -f coral-agents/buyer-agent/Dockerfile  -t buyer-agent:0.1.0  .
docker build -f coral-agents/seller-agent/Dockerfile -t seller-agent:0.1.0 .
```

Then:

```sh
docker compose up -d coral
cd examples/txodds
npm run coral
```

`round.ts` asks the running proxy for a live fixture id from `/api/board` (used only by the
`txline` fallback path), starts the buyer plus three seller personas, and injects `SETTLEMENT_MODE`
+ the `anicca`/ClawRouter/telemetry options above into the agents (see "Settlement in THIS fork"
above for why `direct`, not `arbiter`).

## Watch It

Coral names containers by UUID, so find and tail them:

```sh
docker logs -f $(docker ps -qf ancestor=buyer-agent:0.1.0  | head -1)
docker logs -f $(docker ps -qf ancestor=seller-agent:0.1.0 | head -1)
```

Set `TRACE=1` in `.env` for Coral calls, PDA addresses, and Explorer links.
